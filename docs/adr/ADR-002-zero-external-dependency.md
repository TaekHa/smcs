# ADR-002: 외부 의존성 0 원칙

- **Status:** Accepted
- **Date:** 2026-05-15
- **Author:** Architect Winston (사용자 결정 반영)
- **관련 문서:** PRD NFR12, NFR1, §8 / ARCHITECTURE §3.1, §8

## Context

초기 PRD 초안은 카카오워크/슬랙 알림과 사내 SMTP 이메일 발송을 포함했다. 사용자 검토 결과, 다음 결정이 내려졌다:

- 알림은 시스템 내부에서만 울리면 됨 (외부 메신저 불필요)
- 이메일로도 알림 보내지 않음
- 사내 서버 운영, 개인정보 최소화

이 맥락에서 외부 시스템 의존도를 어디까지 허용할지 결정해야 한다.

## Decision

**SMCS는 어떤 외부 시스템과도 통신하지 않는다.** 폐쇄망에서도 100% 동작한다.

구체적으로:
- ❌ 카카오워크/슬랙 Webhook
- ❌ 이메일 (SMTP)
- ❌ SMS
- ❌ SSO (LDAP/SAML/OIDC)
- ❌ 외부 결제
- ❌ 외부 LLM/AI API
- ❌ 클라우드 스토리지 (S3 등)
- ❌ 외부 APM/모니터링

대안:
- 알림 → **인앱 알림** (DB + 30s polling, ADR-005)
- 인증 → 자체 사용자 + JWT
- 파일 저장 → 로컬 디스크 + Docker volume
- 모니터링 → Spring Actuator + 파일 로그

## Consequences

### 긍정적
- ✅ 사내 폐쇄망에서 완전 동작 — 보안 정책상 외부 통신 불가한 환경에 적합
- ✅ 운영 리스크 감소 — 외부 서비스 장애/토큰 만료/요금 변동 영향 없음
- ✅ 컴플라이언스 단순 — 데이터가 외부로 나가지 않음
- ✅ 개발 범위 축소 — Webhook 통합, SMTP 설정, 외부 인증 연동 작업 불필요

### 부정적
- ⚠️ 외부 메신저 알림이 익숙한 사용자에게 적응 필요
- ⚠️ SSO 미지원 — 사용자가 별도 비밀번호 관리해야 함
- ⚠️ 보고서 이메일 발송 불가 — 매일 시스템에 직접 접속해 확인해야 함

### 진화 경로
- v2 트리거가 발생하면 **어댑터 패턴**으로 추가 (예: `NotificationChannel` 인터페이스에 `KakaoworkChannel` 구현 추가)
- 모듈 경계가 명확하므로 추가 비용 낮음

## Alternatives Considered

### 대안 1: 카카오워크 1개 채널만 통합
- ❌ 기각 — 사용자가 "내부에서만 울리면 됨"으로 명시. 외부 의존성 1개도 운영 부담

### 대안 2: 이메일만 통합 (보고서 발송용)
- ❌ 기각 — 사용자가 "이메일로도 알림 안 가"로 명시
- 시스템 내 보관함으로 대체하면 차별화 가치 유지 가능

### 대안 3: 사내 SMTP는 OK, 외부 SaaS는 불가
- ❌ 기각 — 사내 SMTP도 별도 인프라 가정/유지보수 필요. 사용자의 단순화 선호에 맞지 않음
