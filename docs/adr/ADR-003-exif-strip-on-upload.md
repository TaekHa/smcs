# ADR-003: 이미지 업로드 시 EXIF 자동 스트립

- **Status:** Accepted
- **Date:** 2026-05-15
- **Author:** Architect Winston (사용자 결정 반영)
- **관련 문서:** ARCHITECTURE §6.7, PRD NFR2

## Context

현장 작업자가 폰 카메라로 이슈 사진을 첨부할 때, JPEG는 EXIF 메타데이터(GPS 좌표, 카메라 모델, 촬영 시각 등)를 포함한다. EXIF는 두 측면이 있다:

- **가치:** 작업 위치 추적, 시간 검증, 디바이스 식별
- **위험:** 개인정보(GPS = 현장 작업자 위치) 노출, 디바이스 핑거프린팅

PRD NFR2는 "개인정보 항목 최소화 원칙"을 명시한다. 사용자에게 EXIF 보존/제거 여부를 물은 결과 **자동 스트립**을 선택했다.

## Decision

**모든 업로드 이미지의 EXIF 메타데이터를 서버에서 자동 제거한 후 저장한다.**

구체적으로:
- 라이브러리: **Thumbnailator** 0.4.20
- 처리 시점: Multipart 파싱 직후, 디스크 저장 전
- 모든 메타데이터 제거 (GPS, 카메라, 촬영 시각, 썸네일 등)
- JPEG 품질 95 유지 (시각적 손실 최소화)
- 클라이언트 Canvas 리사이즈도 자연히 EXIF 제거하지만, **defense in depth**로 서버에서 한 번 더 처리

## Consequences

### 긍정적
- ✅ 개인정보 최소화 원칙 충족 — GPS/디바이스 정보 저장 없음
- ✅ 개인정보 유출 사고 시 영향 범위 축소
- ✅ 디바이스 핑거프린팅 차단

### 부정적
- ⚠️ GPS 기반 v2 기능(현장 위치 자동 기록)이 EXIF에서 자동 추출 불가 → **명시적 입력 필드 필요**
- ⚠️ 촬영 시각 손실 — 업로드 시각으로 대체 (큰 차이 없음)
- ⚠️ 처리 비용 — 단, Thumbnailator는 빠름 (1-2MB JPEG 처리 ~50ms)

### 진화 경로
- v2에서 GPS 기능이 필요해지면, 모바일 앱이 **명시적으로** `location_lat`, `location_lng` 필드에 GPS를 첨부 (사용자 동의 후)
- EXIF 보존 정책 재검토는 v2에서 별도 ADR로

## Alternatives Considered

### 대안 1: EXIF 보존
- 초기 후보였으나 사용자가 변경 결정
- ❌ 기각 — 개인정보 최소화 원칙과 충돌. v2에서 명시적 GPS가 더 깔끔한 설계

### 대안 2: 클라이언트 리사이즈만 (서버 스트립 X)
- ❌ 기각 — 사용자가 풀해상도 원본을 의도적으로 업로드하면 EXIF 보존됨. 일관성/보안 결함

### 대안 3: GPS만 제거, 다른 EXIF 보존
- ❌ 기각 — 부분 제거는 복잡도 ↑, 누락 위험 ↑. all-or-nothing이 단순하고 안전
