# Story 1.1: 프로젝트 셋업 및 헬스 체크

## Status: Review

## Story

- As a **개발자**
- I want **Spring Boot 백엔드와 React 프론트엔드를 모노레포로 셋업하고 헬스 체크 엔드포인트를 노출**하기를
- so that **이후 모든 기능 개발의 기반이 마련된다**

## Acceptance Criteria (ACs)

1. `backend/` 디렉터리에 Spring Boot 3.x + Gradle + Java 21 프로젝트가 셋업되어 있다.
2. `frontend/` 디렉터리에 Vite + React 18 + TypeScript 프로젝트가 셋업되어 있다.
3. 백엔드 `GET /api/health` 가 `{ "status": "UP" }` 반환한다.
4. 프론트엔드 `/` 접속 시 "SMCS" 타이틀과 헬스체크 결과를 표시하는 캔버스 페이지가 보인다. **[Story 1.4 에서 LoginView 로 교체되어 superseded. `getHealth()` 호출은 제거됨. 현 상태: `/` 접속 시 인증 가드가 동작하여 로그인 화면 (또는 역할별 메인) 으로 라우팅됨. README 의 동작 검증 섹션 참조.]**
5. Docker Compose로 backend + PostgreSQL을 띄울 수 있다.
6. Flyway 가 셋업되어 있고, 빈 마이그레이션 파일 1개로 부팅 가능하다.
7. README에 로컬 실행 방법이 1페이지로 정리되어 있다.

### API Endpoint 명세

`GET /api/health` — Public (permitAll).

- Request: 없음
- Response 200: `{ "status": "UP" }`
- 인증: 불필요. [Source: architecture/7-api-design.md] — `permitAll: /api/auth/login, /api/health`
- 본 스토리에서 Spring Security 는 아직 도입 전이므로 SecurityFilterChain 설정은 불필요. Story 1.3에서 Security 도입 시 `/api/health` 를 permitAll 처리한다 (이 점을 Story 1.3 의 사전 컨텍스트로 남김).

### 컨테이너 토폴로지 (Story 1.1 한정)

PRD AC 5의 "Docker Compose로 backend + PostgreSQL을 띄울 수 있다" 는 prod 전체 스택(nginx + frontend dist + backend + postgres) 까지를 의미하지 않는다. 본 스토리에서는:

- **PostgreSQL 만 컨테이너로** (`docker/docker-compose.local.yml`)
- 백엔드는 호스트에서 `./gradlew bootRun` 으로 실행 → 컨테이너 PostgreSQL 에 `jdbc:postgresql://localhost:5432/smcs` 로 접속
- prod compose (nginx + backend 이미지 + postgres + secrets) 작성은 Story 4.6 범위. [Source: architecture/10-deployment-architecture.md#10.2]

이 해석은 [Source: architecture/12-developer-experience.md#12.1] "docker compose -f docker/docker-compose.local.yml up -d # postgres만 띄움" 과 정확히 일치한다.

### 코딩 표준 적용 포인트

- 프론트엔드: 컴포넌트에서 직접 `fetch`/`axios` 호출 금지. `src/api/client.ts` 와 `src/api/health.ts` 만이 axios 를 직접 알도록 분리. [Source: prd/9-coding-standards.md#9.1]
- 백엔드 API 경로는 kebab-case (현재 `/api/health` 는 단일 단어라 해당 없음). [Source: prd/9-coding-standards.md#9.2]
- 비밀번호/시크릿 하드코딩 금지: `application-prod.yml.local` 은 `.gitignore` 처리. `application-prod.yml.example` 은 placeholder 만. [Source: prd/9-coding-standards.md#9.1]
- 에러 응답 표준 `{ "code": "...", "message": "..." }` 는 Story 1.3 의 전역 ExceptionHandler 도입 시 적용. **본 스토리에서는 health 컨트롤러가 항상 200을 반환하므로 표준 에러 포맷 작업은 범위 밖.**

### 보안 / 시크릿 (Story 1.1 한정)

- 로컬 DB 비밀번호(`smcs/smcs`)는 dev 전용. `application-local.yml` 에 평문 포함 OK.
- 운영 시크릿 (JWT secret, DB 비밀번호, 데이터 키, HMAC 키 등) 은 Story 4.6 의 Docker secrets 메커니즘으로 관리 예정. [Source: architecture/10-deployment-architecture.md#10.2]
- 본 스토리에서는 보안 모듈 도입 X. Spring Security 의존성도 `build.gradle.kts` 에 **아직 추가하지 않는다** (Story 1.3 범위). 추가하면 디폴트 보안 필터로 인해 `/api/health` 가 401 응답하여 AC 3을 만족시키기 어려워진다.

### 알려진 트랩 / 주의사항

1. **빈 Flyway 마이그레이션 파일**: 완전히 빈 `.sql` 은 Flyway 10.x 에서 허용되지만 검증을 위해 무해 statement(`SELECT 1;` 또는 주석만) 한 줄을 둔다. AC 6 "빈 마이그레이션 파일 1개로 부팅 가능" 의 의도는 "스키마 변경이 없는 상태에서 Flyway 가 OK 마크를 남기고 부팅" 이다.
2. **Vite proxy 설정**: `vite.config.ts` 의 `server.proxy['/api']` 가 누락되면 프론트가 `/api/health` 호출 시 5173 으로 가서 404 → AC 4 실패. 반드시 proxy 설정 검증.
3. **CORS**: 본 스토리에서 백엔드는 Vite 프록시 경유로만 호출되므로 CORS 설정 불필요. 직접 5173 → 8080 호출하면 CORS 실패하니, **상대 경로 `/api/health` 만 사용**한다.
4. **Spring Boot Actuator 충돌 회피**: Actuator 의 기본 health 경로는 `/actuator/health` 이므로 `/api/health` 와 충돌하지 않는다. 다만 PRD AC 가 명시한 응답 본문은 `{"status":"UP"}` — Actuator 기본 본문과 동일하지만 본 스토리에서는 **명시적인 단순 컨트롤러** 로 구현하여 향후 응답 스키마 확장(예: 빌드 정보 포함) 의 자유도를 확보한다.
5. **포트 충돌**: 호스트 5432 가 이미 점유 중이면 docker-compose 가 실패. README 에 "포트 5432가 비어 있는지 확인" 안내.
