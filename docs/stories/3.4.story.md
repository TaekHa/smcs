# Story 3.4: 보고서 자동 생성 스케줄러 + 보관함 적재

## Status: Review

## Story

- As a **관리자**
- I want **매일 아침 시스템에 어제 보고서가 자동으로 준비되어 있기를**
- so that **별도 작업 없이 어제의 상황을 즉시 확인할 수 있다**

## Acceptance Criteria (ACs)

1. 매일 08:00 KST 에 전일 일간 보고서가 자동 생성되어 **시스템 보고서 보관함**에 적재된다(파일 시스템 + DB 메타데이터).
2. 매주 월요일 08:00 KST 에 지난 주 주간 보고서가 자동 생성되어 적재된다.
3. 외부 이메일/메신저 발송은 **하지 않는다**.
4. 동일 일자/주차에 대해 이미 생성된 보고서가 있으면 덮어쓴다(재실행 시 idempotent).
5. 생성 시각은 설정 파일로 변경 가능.
6. 새 보고서 생성 시 ADMIN 사용자에게 **인앱 알림**이 전달된다 ("어제 일간 보고서가 준비되었습니다").
7. 보고서 생성 실패는 로그에 기록되고 관리자에게 인앱 알림으로 전달된다.

2. **[검토] 실패 알림 정책 (AC7)**: 동일 잡이 매일 실패하면 ADMIN 알림 폭주 가능(예: PDF 폰트 에러 → 매일 같은 메시지). **MVP 정책**: 매 실패마다 1회 알림(중복 억제 없음 — 단순). 운영 누적 시 v2 에서 dedup/throttle 도입.
3. **[검토] 스케줄 시각 8:00 KST 고정 → cron 설정화**: AC5 "생성 시각 설정 파일로 변경 가능" 충족 위해 `smcs.reports.daily-cron`/`weekly-cron` 설정 키 신설. 기본 `0 0 8 * * *` (KST) / `0 0 8 * * MON` (KST). zone="Asia/Seoul" 명시.
4. **[검토] 3.3 QA 잔여 관찰 #1 (메모리 위험) 동반 해소**: `ReportService.loadOpenIssues` 페이징 도입은 3.3 코드 수정. **본 스토리 범위 포함 여부 PO 결정** — 포함 시 surgical 변경(Specification + Pageable, footnote 정확성 유지를 위해 `count*` 별도 호출). 미포함 시 별도 tech debt 스토리로 분리.
5. **[기록] notifications 확장 = 분리 테이블 아님**(PO 2026-05-22 비준): 보고서 알림용 별도 테이블 신설 대신 기존 인프라(벨/폴링/리스트) 재사용 — epic3_architecture 명시.
6. **[기록] CHECK 익명명 처리**: V2 `CHECK (kind IN (...))` 익명 → V7 `DROP CONSTRAINT IF EXISTS notifications_kind_check`. Postgres 자동 명명 컨벤션 `{table}_{column}_check` 추정. CI 실행 시 SQL 에러로 즉시 가시화 — 잘못된 이름이면 V7 가 fail-fast 로 차단.
7. **[비준 PO v0.2] 8:00 KST 다운 시 복구 정책 = MVP 무복구**: 시스템 다운으로 cron 잡 미실행 시 그날 보고서 누락 — **자동 catch-up 없음**(운영자 수동 트리거 또는 다음 정상 잡까지 대기). **이유**: MVP 단순성(CLAUDE.md "no speculative code") + 잡 1회/일이라 실제 손실 영향 제한. v2 에서 `@Scheduled` 미스 감지(예: `ShedLock` 또는 마지막 성공 시각 비교) + catch-up 잡 도입 검토. **수동 트리거 권장**: ReportArchiveService 메서드 직접 호출(임시 Actuator 또는 향후 ADMIN 컨트롤러 신설 — 본 스토리 범위 밖).
