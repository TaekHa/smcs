# Story 4.6: 운영 배포 + 백업

## Status: Approved

## Story

- As a **운영자**
- I want **SMCS 를 사내 서버에 안정적으로 배포하고 백업하기를**
- so that **데이터 손실 없이 운영할 수 있다**

## Acceptance Criteria (ACs)

1. **Docker Compose 단일 명령 기동**: `docker compose -f docker/docker-compose.prod.yml up -d` 한 줄로 backend + frontend + nginx + postgres 4개 컨테이너 기동. (PRD §4.6 AC1 "(선택) redis" 는 **MVP 제외** — Deviation #1).
2. **HTTPS 인증서 설정 가이드**: `docs/operations.md` 1페이지에 사내 CA 인증서 적용 단계(`nginx/ssl/server.crt` + `server.key` 마운트) + Let's Encrypt 폴백 방식 명시. 인증서 미설치 시 HTTP 80 → HTTPS 443 redirect 활성/비활성 설정 둘 다 안내.
3. **DB 일일 자동 백업**: `scripts/backup-db.sh`(pg_dump --format=custom 압축) + crontab 등록 가이드. **30일 보관 룰** = 31일 이전 파일 `find ... -mtime +30 -delete`. 백업 파일명 `smcs-db-YYYY-MM-DD.dump`. 백업 저장 위치는 **PO 결정 필요** (Deviation #3).
4. **첨부 파일 디렉토리 백업**: `scripts/backup-files.sh`(`smcs.files.dir` rsync 또는 tar 일자별) + 동일 30일 보관. DB 백업과 별도 스크립트(파일 변동 트랜잭션 무관, rsync 증분 가능).
5. **헬스 체크 통과**: 운영 컨테이너 기동 후 `curl -fsS http://<host>/api/actuator/health` → 200 `{"status":"UP"}` 응답. Docker `healthcheck` 정의로 컨테이너 상태 표시. nginx upstream 헬스체크에 동일 endpoint 활용.
6. **운영 가이드 1페이지** `docs/operations.md`(신규): (a) 서버 재기동 명령 (b) 백업/복원 명령(`pg_restore`) (c) 로그 위치(호스트 `/var/log/smcs/` 마운트 + `docker compose logs`) (d) 환경변수 체크리스트 (e) admin1 초기 비밀번호 변경 의무 (f) 보고서 cleanup-cron 점진 활성화 절차(3.5 lesson) (g) 트러블슈팅(헬스체크 실패/DB 연결 실패/JWT secret 누락).

### Deviation Analysis (PO 비준 필요 v0.1)

1. **[검토 PO] Redis 제외** — PRD §4.6 AC1 "(선택) redis" 명시. 실제 코드 grep 결과 redis 사용 0(`@Cacheable`/`RedisTemplate`/`spring-data-redis` 모두 부재). 폴링 30s + DB 직접 조회로 충분. **MVP 제외, v2 캐시 도입 시 재추가**. → 도커 컴포즈 3 → 4 서비스로 감소(postgres+backend+frontend+nginx).
2. **[검토 PO] nginx 단일 vs 분리** — 옵션 A: 외부 nginx 1개(SSL + reverse proxy + 정적 서빙 통합) → frontend 컨테이너 제거 가능, 빌드 산출물만 외부 nginx volume 마운트. 옵션 B: frontend(nginx 정적) + 외부 nginx(SSL + reverse proxy) 분리 → 컨테이너 2개, 관심사 분리. **B 권장** (각 팀 독립 배포 가능, frontend 빌드 캐시 효율). PO 결정 후 Dev 착수.
3. **[비준 PO v0.2] 백업 저장 위치 = 동일 서버 로컬 `/backup/`** (사용자 결정 2026-05-27). MVP 단순화 우선. ⚠️ **Disaster Recovery 한계 명시**: 서버 디스크 손실 또는 화재/물리 사고 시 DB 백업과 첨부 백업이 *동시 분실* 가능 — 운영 가이드 §4 "백업/복원" 섹션 최상단에 **경고 박스** 필수 + **v2 별도 NAS/S3 sync 강력 권장** footnote 명시. 백업 디렉토리는 별도 디스크/파티션 마운트 권장(`/backup` 을 OS root 와 다른 물리 디스크) — 디스크 단일 장애 일부 보호.
