# Story 4.5: 카테고리 3단계 관리 (Admin)

## Status: Review

## Story

- As a **관리자**
- I want **L1/L2/L3 카테고리를 직접 관리하기를**
- so that **운영 변화에 빠르게 대응할 수 있다 (그리고 Story 4.2 자동 카테고리 제안의 키워드 SoT를 제공한다)**

## Acceptance Criteria (ACs)

1. `/admin/categories` 에서 레벨별 탭(L1/L2/L3)으로 카테고리 리스트가 표시된다. **inactive 포함 전체 표시**(Admin 관리 화면 ↔ 기존 `/api/categories` 는 active-only 유지).
2. 각 레벨에서 **추가 / 이름 수정 / 순서 변경(sortOrder) / 비활성화** 가능. 삭제 = **soft delete (`active=false`)** — 과거 이슈가 FK 참조 중이라 hard delete 금지.
3. 각 카테고리에 **키워드 배열(`keywords: string[]`) 편집** 가능 (Story 4.2 자동 분류 SoT). antd `Select mode="tags"` (Enter/콤마 추가 + 칩 제거).
4. 비활성화된 카테고리는 신규 이슈 등록 폼 드롭다운(`GET /api/categories`)에서 숨겨지지만, 과거 이슈는 영향받지 않는다. 기존 동작 보존 — 4.5는 endpoint 신규만 추가.
5. 시드 데이터(L1: 아파트먼트v1·v2·voip/pbx / L2: 관리자웹·입주민앱·단말·서버 / L3: 기기미동작·기기오동작·로그인오류)는 초기 진입 시 자동 표시(V3 기존, 변경 0).
6. SLA 관리 화면은 **없다** (PRD §5.6 v0.2 제거 — N/A, 4.3 #1 선례).

### Deviation Analysis (PO 비준 완료 v0.2 — Sarah, 2026-05-26)

1. **[비준 PO v0.2] JSONB 매핑 = Hibernate 6 `@JdbcTypeCode(SqlTypes.JSON)`**: AttributeConverter+Jackson 대안 대비 의존성 0 + 보일러플레이트 0. Spring Boot 3.x 네이티브. 통합 테스트가 한글 키워드(`["인터넷","wifi"]`) 라운드트립 검증 의무.
2. **[비준 PO v0.2] POST 단일 upsert (id 유무로 분기)**: PRD §6 line 34 SoT — "POST /api/admin/categories 카테고리 추가/수정". 단일 엔드포인트 비준. **응답 코드 명확화**: id null → 201 Created, id present → 200 OK — 통합 테스트에서 명시 검증. 4.4 사용자 관리도 동일 패턴 채택 후보(향후 일관성).
3. **[비준 PO v0.2] Soft delete only (`active=false`)**: hard delete 금지 — issues 의 category_l1/l2/l3_id FK 참조 + AC4 "과거 이슈는 영향받지 않는다" 명시 의무. UI 에 "삭제" 버튼 부재, 비활성화 Switch + Popconfirm 만. 영구 데이터 정책은 v2 휴지통(soft-then-hard) 검토 가능.
4. **[비준 PO v0.2] 순서 변경 = ↑/↓ 인접 스왑**: 의존성 0 + MVP 사용 빈도 낮음(10행 내외 카테고리). DnD(`@dnd-kit/sortable` 등) 는 v2 — 카테고리 수가 늘거나 UX 피드백 발생 시 도입. mutation 2회(인접 행 sortOrder 스왑) — race 발생 가능성 매우 낮음(ADMIN 단독 화면).
5. **[비준 PO v0.2] 키워드 편집 = antd `Select mode="tags"`**: Enter/콤마 추가 + 칩 제거 — 직관적, 기존 antd 의존성. CSV 입력 form 대안은 UX 열등(파싱 에러 위험). **Story 4.2 자동 분류 SoT 보장** — 본 스토리가 4.2의 키워드 데이터 입력 경로.
6. **[비준 PO v0.2] CategoryOption 공개 DTO 보존**: 폼/필터 사용처(`useCategories` 훅) 변경 0 — 범위 안정. Admin 만 sortOrder/active/keywords 노출(`CategoryAdminResponse`).
7. **[비준 PO v0.2] 캐시 invalidate 2종**: Admin upsert 성공 시 `invalidateQueries(['admin-categories', level])` + `invalidateQueries(['categories', level])` — AC4 "비활성화된 카테고리는 신규 이슈 등록 폼 드롭다운에서 숨겨진다" 즉시 반영 SoT. 누락 시 폼이 5분(staleTime) stale 카테고리 표시 위험.
8. **[비준 PO v0.2] sortOrder 자동 부여**: create 시 미입력 → `max(sortOrder for level) + 1`. 사용자 부담 0, 충돌 가능성 낮음(ADMIN 단독, 같은 level). FE placeholder 만 노출. 명시적 입력 시 그대로 사용(중복 허용 — 동일 sortOrder 는 idAsc 보조 정렬로 안정).
9. **[비준 PO v0.2] 이름 unique 제약 v2**: V2 schema 에 unique constraint 부재 — 동일 레벨/parent 내 중복 가능. 4.5 범위 = 입력 그대로 수용(비즈니스 룰 강제 v2). 운영 중 중복 발생 시 admin 이 직접 정리(soft-rename).
10. **[비준 PO v0.2] parent_id 노출 안함**: V3 시드 모두 parent_id NULL — PRD §5.6 명시 "L1/L2/L3 자유 조합". Admin UI 에서 parent 입력/표시 0 — 4.5 범위 = level 내 평면 관리. v2 매트릭스 룰(특정 조합만 허용) 도입 시 parent_id 활용 검토.
