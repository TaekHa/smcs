# Story 1.2: DB 스키마 및 시드 데이터

## Status: Done

## Story

- As a **개발자**
- I want **핵심 엔티티(User, Issue, Comment, Attachment, IssueEvent, Category(L1/L2/L3), Notification) DB 테이블을 Flyway로 생성하고 시드 데이터를 자동 로딩**하기를
- so that **이후 개발과 테스트에 즉시 사용할 수 있다**

## Acceptance Criteria (ACs)

1. Flyway 마이그레이션으로 7개 테이블이 생성된다: `users`, `issues`, `comments`, `attachments`, `issue_events`, `categories` (self-reference), `notifications`.
2. 외래키, 인덱스가 정확히 설정되어 있다. 특히 `issues(priority, created_at)` 복합 인덱스, `notifications(recipient_id, read_at)` 인덱스 포함.
3. **카테고리 시드 데이터(프로덕션 포함)**: L1 3건(`아파트먼트v1`, `아파트먼트v2`, `voip/pbx`), L2 4건(`관리자웹`, `입주민앱`, `단말`, `서버`), L3 3건(`기기미동작`, `기기오동작`, `로그인오류`).
4. `application-local.yml` 프로파일에서 추가 시드(사용자 8명: AGENT 3 + FIELD 4 + ADMIN 1, 샘플 이슈 20건)가 로딩된다.
5. 프로덕션 프로파일에서는 사용자/이슈 시드는 로딩되지 않는다(카테고리는 로딩됨).
6. `SlaPolicy` 테이블은 만들지 않는다 (v0.2에서 제거).

## Dev Notes

### Previous Story Insights (Story 1.1)

[Source: docs/stories/1.1.story.md#dev-agent-record]

- **Flyway V1 lock**: `V1__init.sql` 은 `SELECT 1;` placeholder 로 prod/local DB 양쪽에 적용되어 checksum 이 이미 기록됨. **V1 을 수정하면 안 되며, 본 스토리는 V2 부터 추가한다.** 만약 어떤 이유로 V1 수정이 불가피하면 `./gradlew flywayRepair` 또는 dev DB 볼륨 초기화(`docker compose -f docker/docker-compose.local.yml down -v`) 필요.
- **JPA ddl-auto 결정**: Story 1.1 에서 미설정 (default `none`). 본 스토리는 JPA 엔티티를 도입하지 않으므로 ddl-auto 효과 없음. **Epic 2 의 첫 엔티티 도입 스토리에서 `validate` 명시 권장** (Flyway 가 스키마 owner — Hibernate 가 schema 를 건드리지 않도록).
- **Spring Security 부재**: Story 1.3 범위. 본 스토리는 BCryptPasswordEncoder 만 `spring-security-crypto` 단독 모듈로 가져온다. **`spring-boot-starter-security` 전체를 지금 추가하면 디폴트 보안 필터로 `/api/health` 가 401 응답** (Story 1.1 AC 3 회귀).
- **`spring-boot-starter-data-jpa` transitive 의존**: `spring-jdbc` (JdbcTemplate) 가 함께 들어옴 → LocalDataSeeder 는 추가 의존성 없이 JdbcTemplate 사용 가능.

### NULLABLE / 컬럼 암호화 결정

[Source: architecture/6-security-architecture-defense-in-depth.md#6.4, #6.5]

architecture §5.1 ER 다이어그램의 `issues.caller_name_enc` / `caller_phone_enc` / `caller_phone_hash` 는 컬럼 암호화 (AES-GCM) 및 HMAC 해시 컬럼이다. crypto 모듈 (`com.smcs.crypto.*`) 은 Story 1.3 이후 범위이며, 본 스토리에는 도입되지 않는다. 따라서:

- **본 스토리의 DDL 은 세 컬럼을 모두 NULLABLE 로 작성** → Story 1.2 의 시드 이슈는 caller 필드 모두 NULL 로 INSERT 가능.
- 운영 시 NOT NULL 강제는 **Epic 2 의 이슈 등록 API 서비스/컨트롤러 계층**에서 입력 검증 (`@Valid`) 으로 처리.
- ER 다이어그램이 nullable 여부를 명시하지 않은 다른 컬럼들의 결정은 위 DDL Spec 표에 표시.

### Categories.parent_id 시드 결정

[Source: prd/5-data-models.md#5.6 "카테고리 조합 규칙(MVP 가정)"]

PRD 는 "L1과 L2, L2와 L3 사이에 계층 종속성을 강제하지 않는다 — 모든 조합 가능" 으로 명시. 따라서:

- 본 스토리의 V3 시드는 **L2 4건과 L3 3건의 `parent_id` 를 NULL 로 둔다**. self-reference FK 컬럼은 schema 차원에서만 보존 (v2 의 계층 강제 도입에 대비).
- 이슈 등록 시 (Epic 2) 의 L1×L2×L3 자유 조합 정책과 일치.

### 시드 비밀번호 정책 예외

[Source: architecture/12-developer-experience.md#12.2 "비밀번호는 모두 `dev1234`"]

architecture §12.2 가 dev 시드 비밀번호를 `dev1234` (7자) 로 명시. PRD §6.9 의 정책 "최소 8자, 영문+숫자" 와 충돌하나, **§6.9 의 정책 검증은 Story 1.3 이후의 사용자 등록 API 가 강제하는 룰이며, 시드는 그 API 를 우회**한다. 본 스토리는 architecture §12.2 명시값 (`dev1234`) 을 따른다. 운영 환경 사용자는 §6.9 정책을 100% 준수해야 함을 LocalDataSeeder 의 INFO 로그에 명시적으로 출력 ("Seed credentials: <user>/dev1234 — NEVER use in production. Real users go through Story 1.3 registration with §6.9 policy.").

### Idempotency 결정

[Source: architecture/12-developer-experience.md#12.2 "매 부팅마다 reset (선택)"]

architecture §12.2 는 reset 을 옵션으로 표기. 본 스토리는 **idempotent (skip-if-exists)** 으로 구현 — `SELECT COUNT(*) FROM users = 0` 이 아닐 때만 시드 실행. 이유:
- 매 부팅 reset 은 개발자가 만든 테스트 데이터(예: 새 이슈/코멘트)를 의도치 않게 삭제할 위험.
- reset 은 명시적 행위(`docker compose down -v` 후 재기동) 로 충분.
- 향후 reset CLI 옵션(`--seed-reset`)이 필요하면 v2 에서 추가.
