# Story 3.3: 일간/주간 PDF 보고서 생성

## Status: Approved

## Story

- As a **시스템**
- I want **일/주간 보고서를 PDF로 생성하기를**
- so that **관리자가 다운로드하거나 (3.5 보관함에서) 미리볼 수 있다**

## Acceptance Criteria (ACs)

1. `GET /api/reports/daily?date=YYYY-MM-DD` 가 해당 일자(KST) PDF를 반환한다.
2. `GET /api/reports/weekly?week=YYYY-Www` 가 해당 주(ISO 주차, KST) PDF를 반환한다.
3. PDF는 회사 로고 영역(MVP는 텍스트만), 표지, 요약 KPI, 카테고리 분포, TOP 5 카테고리, 담당자별 처리량, 미처리 리스트를 포함한다.
4. A4 1-2페이지 분량.
5. 한글 폰트가 정상 임베드된다(나눔고딕 등).
6. PDF 생성에 단위 테스트가 작성되어 있다.

### Deviation Analysis (PO 검토 필요)

1. **[기록] reports 테이블 미사용(3.3=on-demand 생성)**: PO 정정 2026-05-22(Sarah) — `reports` 테이블/메타 적재/idempotent는 3.4(V6) 소유. 3.3은 호출 시 PDF 생성·스트림만(영속 없음). 따라서 3.3 마이그레이션 없음. [Source: prd/5-data-models.md#5.8; docs/stories/3.1.story.md v1.1]
2. **[기록] StatsService range 확장 필요**: 3.1의 period(현재) → 3.3의 임의 [from,to). 3.1 구현 형태에 따라 호출 또는 추출 리팩터. 집계 로직 단일 출처 유지(중복 금지).
3. **[기록] 미처리 리스트 = 현재 스냅샷**: 과거 일자 보고서라도 미처리 리스트/openCount는 **생성 시점 현재** 기준(§5.9 openCount 정의 일치). 아침 운영 관점("아직 안 끝난 건")에서 유효.
4. **[기록] Content-Disposition inline**: 미리보기(3.5 새 탭) 호환 위해 inline. 강제 다운로드는 3.5 프론트 blob 처리.
5. **[기록] X-Accel swap-in 비해당**: §5.8의 X-Accel 최적화는 디스크 파일 서빙(3.4 적재/3.5 보관함) 한정. 3.3 on-demand 생성은 직접 스트림만.
