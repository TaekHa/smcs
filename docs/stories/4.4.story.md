# Story 4.4: 사용자 관리 (Admin)

## Status: Review

## Story

- As a **관리자**
- I want **사용자를 추가/비활성화/역할 변경하기를**
- so that **인사 변경에 대응할 수 있다**

## Acceptance Criteria (ACs)

1. `/admin/users` 화면이 사용자 리스트(active 포함), 추가, 편집, 비활성화 기능을 제공한다(ADMIN 전용).
2. 새 사용자 추가 시 **임시 비밀번호 자동 생성** + create 응답에 **1회 평문 표시** + 화면 모달로 ADMIN 에게 노출(복사 버튼 + "다시 표시되지 않습니다" 경고). 재조회 불가(보안).
3. 비활성화된 사용자는 로그인 불가(`active=false` → SmcsUserDetailsService 가 거부), 기존 데이터는 유지(soft delete — `issues.assigned_to`/`created_by`/`uploader_id` FK 보존).
4. 역할 변경 즉시 DB 반영. **단, 활성 JWT 토큰 내 role 은 stateless 라 다음 토큰 발급(자연 만료 또는 재로그인) 까지 구버전 role 사용 가능** — MVP 한계, v2 토큰 블랙리스트 검토(Deviation #5).
5. **ADMIN 자기 자신 비활성화 차단** — 본인 active 토글 시도 시 백엔드 400 `SELF_DEACTIVATION_FORBIDDEN` + 프론트 Switch disabled + tooltip(시스템 잠금 방지, Deviation #6).
6. **username 변경 불가**(V2 UNIQUE 제약 + 식별자 — MVP). update 요청에 username 필드 부재(또는 무시).
7. **비밀번호 사용자 자가 변경**(잊었을 때 등)은 본 스토리 범위 밖 — v2 별도 endpoint. 본 스토리 = 추가 시 임시 비번 + ADMIN 만 reset 가능(현 범위는 reset 도 미포함 — v2).
8. **임시 비번 표시 누락 시 복구 정책 (PO v0.2 명시화)**: ADMIN 이 TemporaryPasswordModal 을 닫은 후 비번을 기록하지 못한 경우, **MVP 복구 절차** = (a) 해당 사용자 active=false 비활성화 → (b) 동일 username 으로 새 사용자 추가 시 DUPLICATE_USERNAME 400 발생 → (c) 기존 사용자 username 변경은 MVP 불가(AC6) → **유일한 옵션 = 기존 사용자 영구 비활성화 + 다른 username 으로 신규 사용자 생성**. v2 = ADMIN reset 비번 endpoint 도입 권장(차단 아님, 본 스토리 범위 밖 명시).

### Deviation Analysis (PO 비준 완료 v0.2 — Sarah, 2026-05-26)

1. **[비준 PO v0.2] 임시 비밀번호 charset = 영숫자(특수문자 제외) + 길이 12**: 채널 전달 안전성 + 71 bit 엔트로피 충분. SecureRandom + 영숫자(`A-Z a-z 0-9`) 62 chars × 12 자리. v2 에서 사용자 자가 변경 강제 정책 도입 시 charset/길이 강화 검토.
2. **[비준 PO v0.2] update endpoint 분리** `POST /api/admin/users/{id}`: 4.4 create 응답만 `temporaryPassword` 포함 → 4.5 단일 upsert 패턴 부적합. PRD §6 line 31 update endpoint 미명시 → 본 스토리에서 추가. **PRD §6 갱신 follow-up PR 권장**(차단 아님 — 별도 docs 작업).
