# Story 3.5: 보고서 보관함 화면 (미리보기/다운로드)

## Status: Done

## Story

- As a **관리자**
- I want **자동 생성된 일/주간 보고서를 시스템 보관함에서 미리보고 다운로드하기를**
- so that **아침 출근 후 바로 어제 상황을 확인할 수 있다**

## Acceptance Criteria (ACs)

1. `/reports` 화면이 일간/주간 탭으로 분리된 보고서 리스트를 제공한다(최신순).
2. 각 행에는 보고서 일자(또는 주차), 생성 시각, 크기, "다운로드" / "미리보기" 버튼이 있다.
3. 미리보기 클릭 시 PDF 가 브라우저 새 탭에서 열린다.
4. 다운로드 버튼이 작동한다.
5. 90일 이상 지난 보고서는 자동 삭제(저장공간 관리). 일일 정리 잡으로 처리.
6. ADMIN 역할만 접근 가능.

### Deviation Analysis (PO 검토 필요)

1. **[검토] 3.4 QA 잔여 #1 (`runArchive @Transactional` self-invocation) 동반 해소 여부**: 본 스토리는 ArchiveService 손대지 않음(StorageService/Repository/Controller/Scheduler 확장만). 동반 해소 = `@Transactional`을 `generateAndStoreDaily`/`Weekly` public 메서드로 이동(surgical) — PR 단위 1줄 변경. 분리 = 별도 tech debt 스토리. **PO 결정**: 분리 권장(Surgical Changes — 3.5 책임 = 보관함 화면).
2. **[검토] 미리보기/다운로드 endpoint 통합 vs 분리**: 통합(`{id}/file?mode=preview|download`)이 단순. 분리(`{id}/preview` + `{id}/download`)는 의도 명확하나 코드 중복. **권장: 통합**(mode 파라미터).
3. **[검토] cleanup 잡 시각 = 03:00 KST**: 한가한 시간. cron 설정화(`smcs.reports.cleanup-cron`). 권장 그대로.
4. **[검토] retention-days = 90 기본**: epic-3 AC5 명시. 설정화(`smcs.reports.retention-days`).
5. **[기록] `filePath` 미노출 (보안)**: 클라이언트가 직접 file_path 추측·요청 시 traversal 위험 — `{id}` 기반 서빙으로 봉인.
6. **[기록] blob fetch 패턴 (인증)**: native `<a href>`/`window.open(serverUrl)` 은 JWT 미첨부 → 401. 반드시 `apiClient.get(..., responseType:'blob')` 후 `URL.createObjectURL` 사용(AuthImage 패턴 연장, §9.1 apiClient 규약).
7. **[기록] 3.3 on-demand 엔드포인트 공존**: 보관함과 별개 — 의도적. 운영자 수동 API 호출 또는 향후 X-Accel swap-in 대상.
8. **[비준 PO v0.2] cleanup 운영 안전 정책**: cleanup 잡은 파일+DB 동시 자동 삭제 → 잘못된 retention 설정 시 복구 불가. **운영 정책**: (a) 신규 환경에서는 `SMCS_REPORTS_CLEANUP_CRON=0 0 5 31 2 *` 로 잡 비활성화 시작, (b) 1주일 적재 검증 후 활성화, (c) 또는 `retention-days=365` 큰 값 시작 → 점진적 축소. **이유**: MVP 단순성 유지(자동 cleanup 코드는 표준대로) + 운영 단계에서 안전성 확보. v2 에서 휴지통(소프트 삭제) 검토 가능.
