## Story

- As a **CS 접수자/관리자**
- I want **모든 이슈를 한눈에 보고 상태/담당자/날짜로 필터링하며 검색**하기를
- so that **진행 상황을 빠르게 파악할 수 있다**

## Acceptance Criteria (ACs)

1. `/issues` 화면이 페이징 가능한 테이블 뷰를 제공한다(50건/페이지).
2. 컬럼: ID, 제목, **카테고리 L1/L2/L3**, 우선순위, 상태, 담당자, 접수일.
3. 필터: 상태(다중), **카테고리 L1/L2/L3 각각(다중)**, 담당자, 날짜 범위.
4. 검색: 제목/내용/발신자명/전화번호 부분일치. **(→ 아키텍처 보안 SoT 와 충돌. Deviation Analysis #1 / PO 판정 필요. 본 스토리 확정 범위: 제목·내용 부분일치(ILIKE), 전화번호 = HMAC 정확매칭, 발신자명 = MVP 미지원.)**
5. **기본 정렬은 우선순위 내림차순(URGENT→HIGH→NORMAL→LOW) → 접수일 오름차순**. 사용자가 컬럼 헤더 클릭으로 다른 정렬 가능하지만 새로고침 시 기본값 복귀.
6. 우선순위 컬럼은 색상 뱃지(URGENT=red, HIGH=orange, NORMAL=blue, LOW=gray) + 텍스트 라벨로 표시.
7. 상태 컬럼도 색상 + 텍스트 뱃지로 표시.
8. 행 클릭 시 이슈 상세 페이지로 이동.

### Deviation Analysis (PO 검토 필요)

1. **[1급] 검색 범위 — AC4 vs 아키텍처 보안 SoT 충돌**: PRD epic AC4 "발신자명/전화번호 **부분일치**". 그러나 architecture §5.3(PRD FR15 매핑) + §6.5 + ADR-004-hmac-search-hash:
   - 발신자 **이름 = 검색 불가(MVP)** (평문 노출 위험, v2 정확매칭만)
   - 발신자 **전화번호 = HMAC 정확매칭만** (부분/와일드카드 보안상 미지원)
   - 본 스토리 확정: 검색 q = `title/description ILIKE '%q%'` + (q 가 숫자 정규화 시 유효 번호면) `caller_phone_hash = HMAC` 정확매칭. 발신자명 검색 미구현. UI 안내 문구로 사용자 기대 관리.
   - **사유**: 보안/PII 결정(ADR-004 + §6.5)이 epic 문구보다 우선(SoT). 2.1 이 이미 HMAC 인프라 구축. PO 승인 + 후속 PRD epic-2 AC4 문구 정정 권장.
2. **[2급] 리스트 권한 범위**: PRD §6 표는 `GET /api/issues` 를 "인증" 으로 표기. 그러나 §6.3 + 화면 라우팅상 데스크톱 `/issues` 는 AGENT/ADMIN. 본 스토리 `GET /api/issues` = AGENT/ADMIN. FIELD 의 배정-한정 리스트는 Story 2.5 의 `GET /api/me/assigned` 별도. PRD §6 권한 컬럼 정밀화 권장.
3. **[기록] 기본 정렬 구현 주의**: `priority` VARCHAR enum 문자열 정렬은 severity 순서와 불일치 → CASE 기반 명시 정렬 필요(Task 1). architecture §5.2 의 `(priority,created_at)` 인덱스는 문자열 순이라 CASE 정렬을 직접 서빙 못함 — MVP 규모(내부도구, 시드 20+)에서 허용, 성능 이슈 시 v2 에서 severity rank 컬럼/함수 인덱스 검토.
