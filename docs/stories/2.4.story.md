## Story

- As a **CS 접수자/관리자(AGENT/ADMIN)**
- I want **이슈에 담당자를 배정하고 상태를 전이시키기를**
- so that **현장 작업자가 작업을 시작할 수 있다**

## Acceptance Criteria (ACs)

1. 이슈 상세 화면에서 담당자 드롭다운(**FIELD 역할 사용자만**)으로 배정 가능.
2. 배정 시 상태가 자동으로 `ASSIGNED` 로 전이된다.
3. 상태 전이 버튼이 현재 상태에서 **가능한 다음 상태만** 표시된다.
4. 상태/배정 변경은 `IssueEvent` 로 자동 기록된다.
5. 잘못된 상태 전이는 백엔드에서 거부된다(예: `NEW` 에서 직접 `DONE` 불가).

[Source: prd/5-data-models.md#5.2/#5.5; architecture/6-...#6.3; prd/2-requirements.md#FR2]

- **상태 전이(FR2)**: `NEW → ASSIGNED → IN_PROGRESS → DONE → VERIFIED` 단방향 + 일부 역방향(`DONE → IN_PROGRESS` 재오픈). **본 스토리 구현 전이**: `NEW→ASSIGNED`(배정 전용), `ASSIGNED→IN_PROGRESS`, `IN_PROGRESS→DONE`. **2.7 이관**: `DONE→VERIFIED`, `DONE→IN_PROGRESS`(재오픈, reason 필수).
- **§6.3 권한**: 담당자 배정 = AGENT/ADMIN(FIELD ❌) / `NEW→ASSIGNED` = AGENT/ADMIN(배정의 부수효과) / `IN_PROGRESS→DONE` = AGENT/ADMIN/**FIELD 본인배정** / `DONE→VERIFIED` = AGENT/ADMIN(2.7). `ASSIGNED→IN_PROGRESS` 미명시 → AGENT/ADMIN/FIELD본인 적용(Deviation #6).
- **`Issue`**: `assignedTo`(nullable Long), `status`, `resolvedAt`(nullable — `→DONE` 시 설정), `updatedAt`(`@PreUpdate` 로 갱신).
- **`IssueEvent`**: `eventType`(ASSIGNED/STATUS_CHANGED/RESOLVED), `fromValue`/`toValue`(상태 전이 시 이전/다음 status 문자열). actorId = 현재 사용자.
- 시간 ISO-8601, 서버 UTC 저장(2.3 동일 — Instant 직렬화).

### API / 규약

[Source: prd/6-rest-api-spec-요약.md; architecture/7-api-design.md#7.2; architecture/8-sequence-flows.md#8.2]

- `POST /api/issues/{id}/assign` (AGENT,ADMIN) — body `{assigneeId}`. 트랜잭션: 배정 + status(NEW→ASSIGNED) + IssueEvent(ASSIGNED). (Notification = Deviation #1.) 200 + `IssueDetailResponse`.
- `POST /api/issues/{id}/transition` (인증 + 서비스 per-transition authz) — body `{to, reason?}`. 서버 현재 상태 확인 → 유효 전이만 허용(무효 → 409 `INVALID_TRANSITION`, AC5). IssueEvent(STATUS_CHANGED / DONE 시 RESOLVED). 200 + `IssueDetailResponse`.
- 에러 표준 `{code,message,traceId}`: `INVALID_TRANSITION`(409), `INVALID_ASSIGNEE`(400), 기존 `ISSUE_NOT_FOUND`(404)/`ISSUE_FORBIDDEN`(403)/`VALIDATION_FAILED`(400) 재사용.
- `GET /api/*` no-store(§7.3) — 변경 후 프론트는 `['issue', id]` 무효화로 재조회.

### Deviation Analysis (PO 검토 필요)

1. **[2급] Notification 부수효과 미구현**: §8.2 의 배정 시 `notifications(ISSUE_ASSIGNED)` 는 Notification 모듈 부재(V2 테이블만) → 알림 모듈 스토리. 2.3 #4 와 동일 선례. 본 스토리 트랜잭션 = 배정/전이 + IssueEvent 까지.
2. **[1급] 전이 범위 분할 — 2.4 vs 2.7**: 본 스토리는 `NEW→ASSIGNED`(배정)·`ASSIGNED→IN_PROGRESS`·`IN_PROGRESS→DONE`(전진) + state machine 메커니즘 구축. **`DONE→VERIFIED`(검수) 및 `DONE→IN_PROGRESS`(재오픈, reason 필수)는 Story 2.7** 이 전이 맵을 확장. epic 2.4 AC3/AC5(전이 메커니즘) 충족, 검수/재오픈 워크플로우는 2.7 epic 범위와 일치. **PO 비준 필요**.
3. **[2급] `→DONE` 시 `resolvedAt` + `RESOLVED` 이벤트 도입**: §5.5 RESOLVED + resolved_at. 본 스토리가 전이 메커니즘으로 도입하고, **Story 2.6 모바일 "완료처리"(AC6 resolvedAt 기록)가 이를 재사용**. 중복 구현 방지. **PO 비준 필요**.
4. **[2급] 배정 대상 FIELD 검증 위치**: AC1 "FIELD 역할만". **백엔드**가 `role==FIELD && active` 강제(`INVALID_ASSIGNEE` 400) + **프론트** `UserSelect` 가 client-side role 필터(`GET /api/users` 확장 안 함 — 전체 active 반환 유지). 이중 방어.
5. **[기록] `Issue` 엔티티 mutator + `@PreUpdate` 최초 도입**: 현재 getter-only/`@PrePersist` 만 → 상태 변경 위해 `assign()`/`transitionTo()` 도메인 메서드 + `@PreUpdate`(updatedAt 갱신) 추가. 외과적, 기존 로직 무손상.
6. **[기록] 전이 권한 정밀화**: PRD §6 `transition`=coarse "인증". §6.3 은 전이별 권한 명시(`IN_PROGRESS→DONE`=FIELD본인 포함). `ASSIGNED→IN_PROGRESS` 는 §6.3 미명시 → AGENT/ADMIN/FIELD본인 적용 결정. 서비스 레이어 authz(`IssueAccessGuard`) — 2.1/2.2/2.3 §6 정밀화 동일 선례. 추후 PRD §6 권한 주석 정밀화 권장.
7. **[기록] 재배정 의미**: 이미 배정/진행 중 이슈에 재배정 시 — assignee 교체 + ASSIGNED 이벤트, status 가 NEW 가 아니면 status 불변(되돌리지 않음). AC2 는 NEW 에서의 배정에 한정 해석.
8. **[기록] 상세화면 컨트롤 = AGENT/ADMIN 한정 + ConfirmModal 미사용**: FIELD 의 상태 전이(작업 시작/완료)는 모바일(2.5/2.6) — 데스크톱 상세의 배정·전이 컨트롤은 AGENT/ADMIN 에게만 렌더. §9.6.6 ConfirmModal(위험 액션=재오픈/비활성화)은 2.7/admin 스토리에서 도입.
