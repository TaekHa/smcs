## Story

- As a **CS 접수자**
- I want **제목/내용을 입력하면 L1/L2/L3 각각 카테고리가 자동 제안되기를**
- so that **3개 드롭다운을 클릭하는 부담을 줄인다**

## Acceptance Criteria (ACs)

1. 카테고리 마스터의 각 노드(L1/L2/L3)에 키워드 배열이 저장되어 있다 — **Story 4.5에서 SoT 완성**. 본 스토리는 그 데이터를 폼에서 소비.
2. 이슈 등록 폼(`/issues/new`)에서 제목/내용에 키워드가 매칭되면 해당 레벨의 카테고리가 자동 선택된다 — **L1/L2/L3 독립 매칭**(레벨 간 종속성 없음, §5.6 자유 조합 일관).
3. 매칭이 모호한 경우(다중 매칭): **(a) 가장 많은 키워드 매칭 > (b) 가장 긴 키워드 매칭** 순. 동률 시 `sortOrder` 우선(공개 폼 정렬과 일관).
4. 사용자는 자동 선택을 자유롭게 수정할 수 있다 — 한번이라도 사용자가 그 레벨을 직접 변경하면 그 레벨은 더 이상 자동 갱신되지 않는다 (per-level `touched` flag).
5. 룰 기반이며 ML/AI는 사용하지 않는다 — 클라이언트 측 substring 매칭(한글 형태소 분석 미사용).
6. 매칭이 없는 레벨은 자동 선택하지 않고 비워둔다(필수 입력으로 사용자가 직접 선택 — Story 2.1 의 Zod required 보존). **매칭 0 케이스 동작 명세 (PO v0.2)**: 자동 제안 hook 은 매칭 0 일 때 **no-op**(setValue 호출 안 함). 이미 자동 또는 사용자가 채운 값이 있다면 그대로 유지(silent clear 금지) — 텍스트 수정 중 일시적 매칭 미스로 인한 데이터 손실 차단.

### Deviation Analysis (PO 비준 완료 v0.2 — Sarah, 2026-05-26)

1. **[비준 PO v0.2] `IssueFormView` form 별도 `useCategories(1/2/3)` 3 호출 (picker 변경 0)**: picker 단일 책임(표시) + form 단일 책임(business — 자동 제안). 다른 사용처(IssueListView 필터)는 자동 제안 불요 — picker prop 추가 시 회귀 위험. TanStack Query 가 ['categories', level] 키 동일 캐시 공유 → 중복 fetch 0.
2. **[비준 PO v0.2] `CategoryOption` 확장 (필드 +1)**: 단일 DTO + endpoint 시그니처 0. payload 증가 미미(카테고리 10행 × 평균 0~3 키워드 ≈ +500B). 기존 사용처는 `keywords` 미사용 호환(TS optional). v2 PDF/CSV 등 다른 사용처도 단일 DTO 재사용.
3. **[비준 PO v0.2] 디바운스 300ms (IssueListView 일관)**: 한글 IME 조합 중 일시 매칭 회피 + 사용자 입력 흐름 자연. lodash 도입 안 함(자체 setTimeout + cleanup).
4. **[비준 PO v0.2] `touched` 영구 (의도적 변경 보호)**: 한번 사용자가 변경하면 영구 자동 변경 차단. 다시 자동 받고 싶으면 의도적 reset 필요. v2 "자동 제안 재활성화" 버튼 검토 가능. 영구 정책이 silent 자동 덮어쓰기로 인한 사용자 혼란 차단.
5. **[비준 PO v0.2] silent 자동 제안 (MVP 시각 노이즈 회피)**: 자동 선택 시 별도 힌트/라벨 미표시 — 사용자가 자유롭게 변경 가능하니 silent 가 자연스러움. v2 "자동 선택됨 — 변경 가능" 토스트/라벨 검토(사용자 피드백 기반).
6. **[비준 PO v0.2] 매칭 = substring + count/length (AC5 룰 기반)**: 한글 형태소 분석/edit distance 미사용 — 외부 라이브러리 0. ADMIN 이 키워드를 정확히 입력하는 책임(4.5 UI 가 보장). v2 매칭 품질 메트릭 수집 검토.
7. **[비준 PO v0.2] `sortOrder` tiebreaker (자연 정렬)**: candidates 는 백엔드에서 이미 sortOrder asc 정렬 → 동률 시 첫 후보 자연 선택. 명시 정렬 코드 불요(주석으로 의도 표기 권장).
8. **[비준 PO v0.2] 매칭 캐싱 0**: 디바운스 + 10행 × ~3 키워드 = 30 substring 호출(수십 ms) — useMemo 등 추가 최적화 부담 없음.
9. **[비준 PO v0.2] keywords payload 다른 사용처 무관**: IssueListView 필터·CategoryPicker 등 TS interface `keywords?: string[]` optional 처리. 영향 0.
10. **[비준 PO v0.2] 자동 제안 = `IssueFormView` 전용**: 모바일 폼(`MobileFieldDetailView`)은 카테고리 변경 0 — 자동 제안 비대상. v2 모바일 폼 추가 시 hook 재사용 가능.
