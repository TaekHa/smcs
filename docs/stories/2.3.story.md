## Story

- As a **사용자(AGENT/ADMIN/배정된 FIELD)**
- I want **이슈의 모든 정보, 활동 이력, 코멘트를 한 화면에서 보고 상호작용**하기를
- so that **별도 도구 없이 협업할 수 있다**

## Acceptance Criteria (ACs)

1. `/issues/:id` 가 이슈 메타데이터, 본문, 첨부 이미지, 활동 로그, 코멘트 섹션을 표시.
2. 상태 흐름이 상단에 가로 진행 바로 시각화된다.
3. 코멘트 작성 폼이 제공되고, 저장 시 활동 로그에 자동 반영된다.
4. 활동 로그가 시간 역순으로 표시(최신이 위).
5. 첨부 이미지 썸네일을 클릭하면 원본 크기 모달이 열린다.
6. 권한 없는 사용자는 편집 액션이 비활성화된다.

### API / 규약

[Source: prd/6-rest-api-spec-요약.md; architecture/7-api-design.md#7.1/#7.2/#7.3]

- `GET /api/issues/{id}` (인증), `POST /api/issues/{id}/comments` (인증), `GET /api/issues/{id}/events` (인증) — PRD §6 표 정의. **코멘트 목록 전용 GET 엔드포인트는 PRD §6 에 없음** → 코멘트는 `IssueDetailResponse.comments[]` 에 임베드, 활동 로그만 별도 `GET .../events`.

### Deviation Analysis (PO 검토 필요)

1. **[1급] `GET /api/issues/{id}` 권한 정밀화 — PRD §6 vs §6.3/§9.5**: PRD §6 표는 coarse "인증". 그러나 §6.3(FIELD 본인 배정만) + §9.5(미달 403). 본 스토리: `isAuthenticated()` + **서비스 레이어 소유권 검증**(@PreAuthorize 로 "본인 배정" 표현 불가). FIELD 비배정 → 403. 2.1(`GET /api/categories`)·2.2(`GET /api/issues` 권한·`GET /api/users`) §6 정밀화 선례 동일 — **PRD §6 `GET /api/issues/{id}`·`/comments`·`/events` 권한 주석 정밀화 권장**(차단 아님).
2. **[1급] 발신자 PII 표시 = AGENT/ADMIN 평문, FIELD null (사용자 결정)**: 2.1/2.2 가 의도적으로 응답에서 발신자 평문 제외하고 "상세 = Story 2.3 결정" 명시. 사용자(팀) 결정: **CS 콜백 목적상 AGENT/ADMIN 에 복호화 평문 표시, FIELD(담당자)에는 미표시(개인정보 최소화)**. §6.4 복호 가능, §6.9 마스킹은 *로그* 한정(화면 표시 별개). **PO 비준 필요** + 추후 PRD 데이터모델/§6.9 주석 정합 권장.
3. **[2급] 첨부 = 표시 UI만, 업로드/서빙 2.6 이관 (사용자 결정)**: 이미지 업로드(§8.3)·보안 서빙 X-Accel-Redirect(§8.6)는 Story 2.6/배포 인프라. 본 스토리는 Attachment **읽기 매핑 + 상세 DTO 노출 + 갤러리/모달 UI**만; 업로드 경로 부재로 **목록 항상 비어있음**. AC1/AC5 의 첨부 부분 = 컴포넌트 테스트(mock)로 검증, 실제 e2e 는 Story 2.6. **PO 비준 필요**.
4. **[2급] `POST .../comments` Notification 부수효과 미구현**: §7.2 는 "관련자 Notification" 명시하나 Notification 엔티티/모듈 부재(V2 테이블만). 알림 생성(배정/코멘트/전이/재오픈 횡단)은 알림 모듈 스토리(2.5 영역/후속)에서 일괄. 본 스토리 트랜잭션 = Comment + IssueEvent(COMMENTED). PO 검토.
5. **[기록] AC6 해석 고정**: 2.3 의 역할 제한 편집 액션 = **코멘트 작성뿐**(§6.3 코멘트=AGENT/FIELD/ADMIN 전원 ✅). 배정/상태전이 컨트롤은 Story 2.4. 따라서 AC6 = (i) 조회 불가자(FIELD 비배정/미인증) 백엔드 403 + 프론트 Forbidden, (ii) 코멘트 폼은 조회 가능자에게만, (iii) 배정/전이 버튼은 2.3 에서 미렌더(2.4). QA 는 "배정/전이 버튼 부재" 를 fail 처리 금지.
6. **[기록] 코멘트 kind = NOTE**: 데스크톱 상세 작성 코멘트는 `NOTE`. `FIELD_ACTION`=Story 2.6 모바일 조치, `SYSTEM`=자동.
7. **[기록] 활동 로그 vs 코멘트 섹션 분리**: 활동 로그 = `IssueEvent`(CREATED/COMMENTED/STATUS_CHANGED/ASSIGNED/RESOLVED/ATTACHMENT_ADDED) `GET .../events` 역순. 코멘트 섹션 = `Comment.body` (DetailResponse 임베드). **AC3 = 코멘트 저장 시 IssueEvent(COMMENTED) 동시 기록 → 활동 로그 자동 반영**(프론트는 두 쿼리 무효화).
8. **[기록] 라우트 가드**: `/issues/:id` = `RequireAuth` only(RequireRole 미추가 — FIELD 담당자 허용). 백엔드가 소유권 403 강제. routes.tsx placeholder 주석 현행화(외과적).
