# Story 1.3: JWT 인증과 역할 기반 권한

## Status: Review

## Story

- As a **사용자**
- I want **사용자명/비밀번호로 로그인하여 JWT 토큰을 받고, 내 역할에 따라 화면 접근이 제한**되기를
- so that **보안과 적절한 권한 분리가 보장된다**

## Acceptance Criteria (ACs)

1. `POST /api/auth/login` 이 username/password로 JWT를 발급한다.
2. JWT에 `sub=userId`, `role`, `iat`, `exp` 클레임이 포함된다 (만료 8시간).
3. 비밀번호는 BCrypt(cost 10) 해시로 저장된다. (Story 1.2의 시드 사용자 비밀번호 검증 통과 = `dev1234` BCrypt 매칭)
4. `GET /api/me` 가 JWT를 검증하고 현재 사용자 정보를 반환한다.
5. Spring Security가 `@PreAuthorize("hasRole('ADMIN')")` 등 메서드 보안을 지원한다.
6. 인증 실패 시 401, 권한 부족 시 403을 **표준 에러 포맷** (`{ code, message, traceId }`) 으로 반환한다.
7. **로그인 Rate Limiting:** 동일 username **또는** 동일 IP 기준 `/api/auth/login` 실패 5회 누적 시 10분 lockout. 성공 시 카운터 초기화. 6회째 시도는 `423 Locked` 응답.
8. **일반 API Rate Limiting:** 동일 사용자(JWT `sub`) 기준 `/api/*` 분당 300 요청 제한. 초과 시 `429 Too Many Requests` + `Retry-After` 헤더.

### Previous Story Insights (Story 1.2)

[Source: docs/stories/1.2.story.md#dev-agent-record]

- **BCryptPasswordEncoder bean 재사용**: Story 1.2 에서 `spring-security-crypto` 단독 의존성으로 BCryptPasswordEncoder 가 도입됨. 본 스토리에서 `spring-boot-starter-security` 가 들어오면 transitive 로 동일 모듈이 포함되어 충돌 없음. SecurityConfig 에서 `@Bean PasswordEncoder` 로 명시 등록 → AuthenticationManager 가 자동 사용.
- **시드 사용자 비밀번호 정책 우회 주의**: Story 1.2 의 LocalDataSeeder 가 `dev1234` (7자) 를 BCrypt 해시로 저장함. PRD §6.9 의 "최소 8자, 영문+숫자" 정책은 본 스토리의 인증 흐름에서 **검증하지 않는다** (정책 검증은 사용자 등록/변경 API 시점에 적용 — Epic 4 범위). 본 스토리는 시드 사용자로 로그인이 동작하는지만 검증.
- **JPA ddl-auto 미설정 → 본 스토리에서 `validate` 명시**: Story 1.2 는 JPA 엔티티 없이 raw JdbcTemplate 사용했으므로 ddl-auto 효과 없었음. 본 스토리에서 `User` 엔티티를 처음 도입하므로 **반드시 `validate`** 설정 (Hibernate 가 Flyway 스키마를 건드리지 않도록).
- **Testcontainers 로컬 실행 제약**: Story 1.2 의 Testcontainers 통합 테스트가 Windows + Docker Desktop 환경에서 docker-java 호환성 이슈로 미실행. 본 스토리의 통합 테스트도 동일 환경에서는 미실행 예상 → **로컬 검증은 Task 14 수동 골든 패스로 갈음**, CI(Linux runner) 도입 시점에 자동 실행. 통합 테스트 코드는 `./gradlew compileTestJava` 통과로 품질 보증.
- **Flyway V1 ~ V3 lock**: V4 가 본 스토리의 신규 마이그레이션. V1/V2/V3 는 수정 금지 (checksum 기록됨).

### JWT 클레임 스펙

[Source: architecture/6-security-architecture-defense-in-depth.md#6.2]

```json
{
  "sub": "5",              // userId (string-cast for JJWT standard)
  "role": "AGENT",         // "AGENT" | "FIELD" | "ADMIN"
  "iat": 1747526400,       // epoch seconds
  "exp": 1747555200        // iat + 28800 (8h)
}
```

- **`sub` 는 string 으로 저장**하되 서버 측에서 Long.parseLong 으로 복원. JJWT 0.12.x 의 `claims.getSubject()` 는 String 반환.
- **`role` 은 단일 String** (`AGENT|FIELD|ADMIN`). 다중 권한이 필요해지면 v2 에서 `roles` 배열로 확장.
- JJWT 직렬화 시 `Jwts.builder().subject(userId.toString()).claim("role", role.name()).issuedAt(now).expiration(exp).signWith(key).compact()`.

### Spring Security 권한 표기 규약

[Source: architecture/6-security-architecture-defense-in-depth.md#6.3]

- DB 의 `users.role` 컬럼 값: `AGENT|FIELD|ADMIN` (Story 1.2 V2 의 CHECK 제약).
- Spring Security 의 `hasRole("ADMIN")` 은 내부적으로 `ROLE_ADMIN` authority 를 찾음 → **SmcsUserDetailsService 에서 `new SimpleGrantedAuthority("ROLE_" + role.name())`** 으로 변환 필수.
- `@PreAuthorize` 표기: `hasRole('ADMIN')`, `hasAnyRole('AGENT', 'ADMIN')` 등 표준 SpEL.

### Completion Notes List

- **AC 1~8 모두 통과** — Task 14 수동 골든 패스로 검증. `agent1/dev1234` 로그인 → JWT 발급, `/api/me` 동작, 401/423/429 표준 에러 포맷, `/api/health` 회귀 없음.

- **JWT 알고리즘 deviation (HS256 → HS512)**: JJWT 0.12.x 의 `signWith(SecretKey)` 가 키 길이에 따라 HMAC 알고리즘을 자동 선택. dev 시크릿(64 bytes) → HS512. architecture §6.2 명시는 HMAC-SHA256 이나 HS512 가 더 강력하므로 보안상 개선. Prod 시크릿도 32+ bytes 권장이라 동일 동작 예상. **Story 1.4 영향 없음** (프론트는 알고리즘 무관, 토큰을 헤더로만 전달).

- **Rate limiting deviation (Bucket4j Spring Boot Starter → 직접 구현)**: 스토리는 `bucket4j-spring-boot-starter 0.12.8` + Caffeine 캐시를 명시했으나, 해당 starter 가 Bucket4j 8.x 패키지 재구조화 (`io.github.bucket4j.caffeine.CaffeineProxyManager`) 와 자동 구성 호환성 이슈 (`No Bucket4j cache configuration found - cache-to-use: caffeine`). **`bucket4j-core` 직접 사용** + `RateLimitFilter` (OncePerRequestFilter, `ConcurrentHashMap<Long, Bucket>`) 로 단순화. 동일 정책(300/min per user) 충족. **Story 1.4 영향 없음.** v2 분산 환경 전환 시 Redis 기반 ProxyManager 로 교체 가능.

- **Filter 자동 등록 함정**: `@Component`(또는 `@Bean`) Filter 는 Spring Boot 가 자동으로 servlet 전역 filter 로 등록. Spring Security 체인 외부에서 먼저 호출되며 `OncePerRequestFilter` 가 security-chain 호출을 차단해 SecurityContext-의존 필터가 동작 안 함. **`FilterRegistrationBean(enabled=false)` 로 자동 등록 명시적 비활성화** 패턴 채택. JwtAuthFilter, RateLimitFilter 모두 적용. Epic 2+ 의 새 필터 추가 시 동일 패턴 권장.

- **Testcontainers 통합 테스트 실행 환경**: Story 1.2 와 동일하게 본 Windows + Docker Desktop 환경의 docker-java 호환성 이슈로 로컬 실행 미수행. 통합 테스트 코드 5건(AuthIT, MeIT, RateLimitIT, MethodSecurityTest, JwtServiceTest, LoginAttemptServiceTest) 모두 `compileTestJava` 통과. **CI(Linux runner) 셋업 시 자동 실행 검증 필요** — Story 1.2 Completion Notes 의 동일 deferral 적용.

- **Story 1.4 (로그인 화면) 에 영향**:
  1. `/api/auth/login` 요청 본문: `{username, password}` JSON. 응답: `{token, expiresInSeconds, user:{id, username, displayName, role}}`.
  2. `/api/me` 응답: `{id, username, displayName, role}` (`UserSummary`). Frontend `AuthProvider` 에 매핑.
  3. 표준 에러 포맷: `{code, message, traceId}`. 프론트 에러 핸들러는 `code` 로 분기 추천:
     - `INVALID_CREDENTIALS` (401) — "사용자명/비밀번호 확인"
     - `ACCOUNT_LOCKED` (423) — "10분 후 다시 시도" UX
     - `UNAUTHORIZED` (401, SecurityConfig entryPoint) — 토큰 만료/누락 시 자동 로그아웃 트리거
     - `RATE_LIMIT_EXCEEDED` (429) — Retry-After 헤더로 backoff
  4. JWT alg=HS512 (위 deviation). 프론트는 알고리즘 직접 검증 안 함, 토큰을 그대로 `Authorization: Bearer` 헤더에 첨부만 하면 됨.
