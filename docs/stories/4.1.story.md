# Story 4.1: 인앱 알림 시스템 (Bell 드롭다운 + 90일 자동 삭제)

## Status: Review

> 신설 배경: PRD §4.1 의 8개 AC 중 AC1·3·4·5·6·7 은 **Story 2.8(Done)** + **Story 2.7(Done, ISSUE_REOPENED)** + **Story 3.4(Done, REPORT_READY/FAILED V7)** 가 이미 충족. Epic 4 진입 시점의 **진짜 잔여 = 2건**: **AC2 (벨 클릭 드롭다운, 최근 10건 + "모두 보기")** + **AC8 (90일 자동 삭제 스케줄러)**. 본 스토리는 Δ-스토리로 2.8 인프라(useUnreadCount/useNotifications/useMarkRead/NotificationsView·routes)를 그대로 재사용하여 두 잔여만 추가한다.

## Story

- As a **시스템 사용자(AGENT/FIELD/ADMIN) + 운영자**
- I want **벨 클릭으로 최근 10건을 드롭다운에서 빠르게 훑고, 오래된 알림은 자동 정리되기를**
- so that **풀-페이지 이동 없이 알림을 처리하고, notifications 테이블이 무한 누적되지 않는다**

## Acceptance Criteria (ACs — PRD §4.1)

1. 헤더 우측 벨 아이콘 + 미읽음 카운트 뱃지. — **2.8 충족(`NotificationBell`)**, 회귀 무손상.
2. **벨 클릭 시 최근 10건 드롭다운**이 열린다. **"모두 보기"** 풋터는 `/notifications` 이동. — **본 스토리 잔여**
3. 알림 항목 클릭 시 해당 이슈 상세로 이동 + `readAt` 자동 갱신. 보고서 알림(`issueId==null`)은 `/reports` 이동(3.4 분기). — **2.8/3.4 충족**, 드롭다운 동작에 동일 적용.
4. `/notifications` 페이지 + "모두 읽음 처리" 버튼. — **2.8 충족(`NotificationsView`)**.
5. 트리거: `ISSUE_ASSIGNED`/`ISSUE_COMMENTED`/`ISSUE_STATUS_CHANGED`/`ISSUE_REOPENED`. — **2.8(3종) + 2.7(REOPENED)** 충족.
6. 30초 주기 `GET /api/notifications/unread-count` 폴링, **사운드/시각 표시 없이 조용히 갱신**(MVP). — **2.8 충족(`refetchInterval:30_000` + `refetchIntervalInBackground:false`)**.
7. 외부 채널(메신저/이메일/SMS) 발송 없음. — 자명(ADR-002/NFR12, 구현 부재로 충족).
8. **알림은 90일 이후 자동 삭제**. — **본 스토리 잔여**

### Deviation Analysis (PO 검토 필요)

1. **[기록] 펄스 애니메이션 미적용**: front-end-spec §6.10 의 "Q5=b 펄스 0.6초" 는 PRD AC6 "사운드/시각 표시 없이 조용히 갱신"(MVP)과 충돌 → **PRD 채택**. v2 후속. PO 비준(AC 우선) 자명.
2. **[기록] Cleanup seq scan 수용**: V2 인덱스(`(recipient_id, created_at)` 다컬럼)는 `WHERE created_at < ?` 단독 쿼리에 미적중. 단일 컬럼 인덱스 V8 추가는 **deferred**(MVP 규모 + 일배치 03:30 KST 단일 실행, 부하 비현실). 알림 누적이 100만 행 초과 시 v2 에서 인덱스 추가 검토. **Deviation 기록만**(차단 아님).
3. **[1급] 드롭다운 데이터 소스 = 기존 list hook 슬라이스**: §8.5 의 `?page=0&size=10` 전용 호출 대신 2.8 `useNotifications(0)`(size=20) 결과의 `slice(0, 10)`. 별도 hook/캐시 분기 회피(SoT 단일 — 드롭다운↔페이지 일관성). **PO 비준 필요**(UX/캐시 결정).
4. **[기록] 클릭 동작 헬퍼 추출(권장)**: `NotificationsView.open()` 로직과 동일 분기(보고서 알림 = `/reports`) — `features/notifications/openNotification.ts` 단일 SoT 권장. 인라인 복제도 허용(Dev 판단 — Task 1 본문에 명시).
5. **[기록] 스케줄러 cron = 03:30 KST**: ReportScheduler cleanup 03:00 + 30분 — 동시 부하 회피. `smcs.notifications.cleanup-cron` 으로 외부화(prod 오버라이드 가능).
6. **[기록] 90일 retention 정책 = 전체 알림 동일**: 이슈 알림(2.8) + 보고서 알림(3.4) 모두 동일 90일. architecture §5.5 표 정합. 분리 정책 v2 트리거.
