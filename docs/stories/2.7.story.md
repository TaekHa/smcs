# Story 2.7: 검수(VERIFIED) 및 재오픈 워크플로우

## Status: Review

## Story

- As a **CS 접수자/관리자(AGENT/ADMIN)**
- I want **현장 작업자가 완료(DONE)한 이슈를 검수(VERIFIED)하거나 재오픈(IN_PROGRESS)할 수 있기를**
- so that **품질이 보장된다**

## Acceptance Criteria (ACs)

1. DONE 상태 이슈에서 ADMIN/AGENT 가 `VERIFIED` 로 전이 가능.
2. 재오픈 시 사유 코멘트 필수 입력.
3. 재오픈된 이슈는 `IN_PROGRESS` 로 돌아가고 활동 로그에 사유가 남는다.

### Deviation Analysis (PO 검토 필요)

1. **[기록] 전이 맵 확장**: DONE→{VERIFIED, IN_PROGRESS}. FR2/epic 2.7 정합.
2. **[1급] DONE 출발 전이 = AGENT/ADMIN 전용**: 2.4 transition 은 owner(FIELD 본인)도 허용(ASSIGNED/IN_PROGRESS 출발 작업). 검수/재오픈은 §6.3·epic 상 AGENT/ADMIN 만 → `from==DONE && !privileged → 403`. 2.4 Deviation #6(전이 권한) 정밀화. **PO 비준**.
3. **[1급] 재오픈 사유 = transition 트랜잭션 내 NOTE 코멘트**: reason 필수(공백→400). 사유를 Comment(NOTE)+COMMENTED 이벤트로 기록(IssueEvent 자유텍스트 없음) → AC2/AC3 충족. 원자적(2.6 의 2-call 과 달리 reason 이 전이에 통합 — 재오픈은 reason 이 본질). **PO 비준**.
4. **[기록] ISSUE_REOPENED 알림**: 2.8 NotificationService.onReopened 추가. 재오픈만 REOPENED, 검수는 STATUS_CHANGED.
5. **[기록] 재오픈 시 resolvedAt 클리어**: 재오픈된 이슈는 완료 아님 → resolvedAt=null. Issue.transitionTo(IN_PROGRESS) 확장.
6. **[기록] 프론트 재오픈 모달**: Ant Modal 사유 입력(§9.6.6 ConfirmModal 취지, 위험액션). 검수는 직접 버튼. AGENT/ADMIN 한정(canManage 2.4).
